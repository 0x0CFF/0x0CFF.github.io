一、文件路径

在计算机中要找到一个文件，就需要知道文件所在的位置，这时候就需要用到文件路径

文件路径用于描述文件在硬盘上的位置，有了文件路径，系统和软件就可以找到目标文件并对其中的数据进行读写

文件路径有两种表示方式：绝对路径、相对路径

绝对路径

绝对路径描述了目标文件在硬盘上的真实位置，绝对路径以系统的根文件夹为开头

  • Window 系统中以盘符（比如 C:\、D:\）作为根文件夹
  • macOS、Linux 系统中以 / 作为根文件夹

举两个例子：

  • 在 Windows 系统中，D:\work\test.txt 是一个绝对路径，它表示 D 盘下有一个 work 文件夹，文件夹中有一个 test.txt 文件

  • 在 Linux 系统中，/home/work/test.txt 是一个绝对路径，它表示家目录 /home 下有一个 work 文件夹，文件夹中有一个 test.txt 文件

相对路径

在使用相对路径之前，需要先知道当前工作目录（也称为前工作路径），工作目录描述了程序运行时的路径

举个例子，在 Linux 系统中，我在 /home/suface/documemt 目录下运行了一个终端程序，此时我们就可以说该终端程序的工作目录为/home/suface/documemt（ pwd 命令用于查看当前终端的工作目录）

知道了当前工作目录后，就可以使用相对路径了

相对路径描述了目标文件相对于工作目录的位置，相对路径以一些特殊标识开头

标识表示
./工作目录
../工作目录的上级文件夹
../../工作目录的上上级文件夹

相对路径使用示例，假设我们在 /home/suface/documemt 目录下运行了一个终端程序，此时该终端程序的工作目录为/home/suface/documemt，则其他文件夹、文件的相对路径表示如下

📁 /                     # 工作目录的上上上级目录，用 ../../../ 表示
└─ 📁 home               # 工作目录的上上级目录，用 ../../ 表示
   ├─ 📁 surface         # 工作目录的上级目录，用 ../ 表示 
   │  ├─ 📁 documemt     # 工作目录，用 ./ 表示
   │  │  └─ 📃 01.txt    # ./01.txt
   │  └─ 📁 other        # ../other
   │     └─ 📃 02.txt    # ../other/02.txt
   ├─ 📁 src             # ../../src
   │  └─ 📃 03.txt       # ../../src/03.txt
   └─ 📃 04.txt          # ../../04.txt

二、打开文件

在终端中使用 hx <绝对路径> 或 hx <相对路径> 命令即可在 Helix 中打开目标文件，以下面的文件树状图为例

# Linux 系统

📁 /                     # 工作目录的上上上级目录，用 ../../../ 表示
└─ 📁 home               # 工作目录的上上级目录，用 ../../ 表示
   ├─ 📁 surface         # 工作目录的上级目录，用 ../ 表示 
   │  ├─ 📁 documemt     # 工作目录，用 ./ 表示
   │  │  └─ 📃 01.txt    # ./01.txt
   │  └─ 📁 other        # ../other
   │     └─ 📃 02.txt    # ../other/02.txt
   ├─ 📁 src             # ../../src
   │  └─ 📃 03.txt       # ../../src/03.txt
   └─ 📃 04.txt          # ../../04.txt    

# Linux 系统

# 绝对路径示例
# 用 Helix 打开 /home/surface/documemt/ 下的 01.txt 文件
hx /home/surface/documemt/01.txt

# 相对路径示例（假设工作目录为 /home/suface/documemt）
# 用 Helix 打开 /home/surface/documemt/ 下的 01.txt 文件
hx ./01.txt
# 用 Helix 打开 /home/surface/other/ 下的 02.txt 文件
hx ../other/02.txt
# 用 Helix 打开 /home/src/ 下的 03.txt 文件
hx ../../src/03.txt
# 用 Helix 打开 /home/ 下的 04.txt 文件
hx ../../../04.txt

打开文件后，就会进入一个紫色的终端程序，这便是 Helix 编辑器

注意：如果目标文件不存在，则会打开一个空白文件，（本质是 Helix 在内存中创建并打开一个了空白文件），只有在 Helix 中执行保存操作时，数据才会写入到硬盘中

四、模式简介

进入 Helix 后，相信初学者很快就会发现一个很奇怪的地方，就是无法直接编辑文字，这是因为 Helix 有两种主要模式：正常模式、编辑模式

正常模式

正常模式用于执行一些快捷操作（比如剪切粘贴、文本替换、撤销重做等等），在正常模式下还存在一些微模式（Minor Mode）

目前 Helix 有以下几个微模式（关于微模式在后面章节会逐个进行讲解）

微模式热键作用
Goto Mode（跳转模式）g操作光标跳转
Match Mode（匹配模式）m匹配文本
Command Mode（命令模式）:执行命令
View Mode（视图模式）z操作视图
Select Mode（选择模式）v选择内容
Window Mode（窗口模式）Ctrl w操作窗口
Space Mode（空格模式）空格键一些无法归类的命令集合

打开微模式后，按 Esc 键可以关闭微模式

编辑模式

编辑模式用于进行文字编辑，在打开一个文本时会自动进入正常模式（Normal Mode），正常模式只负责命令操作，所以敲不出文字

手动按 i 键可以由正常模式切换到编辑模式（Insert Mode），左下角的模式标识符会由 NOR 变为 INS

在编辑模式下，可以正常进行码字，按 Esc 键可以返回正常模式，左下角的模式标识符会由 INS 变为 NOR

注意事项

  • 虽然编辑模式也是在正常模式下通过热键激活，但是编辑模式不属于微模式，它是一个独立的模式
  • 所有的微模式都只能在正常模式下调用，在编辑模式下无法调用

五、疑惑 & 解答

为什么要设计这么别扭的使用逻辑？这么别扭的编辑器真的是给人用的吗？Helix 值得去学吗？刚接触终端文本编辑器的用户大概都会有这样的疑问，之所以这样设计，有这样的几个原因

早期的计算机是没有鼠标和图形界面的，几乎所有的操作都要通过键盘和终端来实现，而文本编辑本身并不止输入和删除文本这么简单，还可能涉及到全选、替换等复杂操作，但是键盘上的按键十分有限，所以终端文本编辑器就设计成多模式的使用逻辑，将文本编辑独立出来，其他模式用于执行命令操作

现代社会依旧有大量的服务器在工作，这类机器的硬件资源比较有限，所以一般不会安装图形界面，开发人员通过终端来进行开发调试，有时碰到需要编辑文本的情况，就需要使用到终端文本编辑器

现代计算机有了鼠标和图形界面，也有了图形化的文本编辑器，更多的功能可以通过鼠标点击菜单来实现，但是对于程序员而言，双手更多时间是在键盘上，离开键盘去握住鼠标再去进行点击操作是一个低效率的行为，同时码字过程中的思绪也容易被这一举动打断，相对而言这类双手不用离开键盘的终端文本编辑器更能满足码字的畅快感

简单来讲就是，终端文本编辑器依旧有它的适用范围，虽然刚开始上手会比较别扭，但是熟悉以后码字效率会非常高，并且码字体验会非常畅快
