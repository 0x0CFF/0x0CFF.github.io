Linux 发行版

由于 GNU 是完全自由的，任何人都可以获取 GNU 项目的源码，并且在遵守 GPL 协议的情况下可以对其进行修改、分发，因此很多商业公司和团队便专门来开发自己的 Linux 发行版，于是诞生了各式各样的 Linux 发行版，比如 Red Hat、SuSE、Debian、CentOS、Arch 等等， GNU / Linux 开始开枝散叶~

主流发行版

Red Hat

红帽公司（Red Hat）是一家开源解决方案供应商，旗下有多个 Linux 发行版

官网：https://www.redhat.com/zh

  • RedHat Enterprise Linux

    简称 RHEL，由红帽公司维护，是面向企业的 Linux 发行版，开源免费使用，但企业可以花费购买升级服务和有偿技术支持

  • CentOS

    RHEL 的克隆版本，更新较慢，具有稳定、长期的升级支持、性能稳定等特点，被企业普遍使用（2020 年 12 月 08 日 CentOS 官方宣布 CentOS Linux 项目将停止）

  • Fedora

    红帽公司的试验田，新的特性会在 Fedora 优先发布，不断优化测试，等到新特性稳定后就加入到 RHEL 中（Fedora 是 RHEL 的上游），由开源社区维护

SuSE

SuSE 公司是全球首家推出企业级 Linux 发行套件的提供商

  • openSUSE

    由 Novell 公司赞助成，致力于为所有人提供一个最易于使用的 Linux 发行版

Debian

Debian 是目前世界最大的非商业性 Linux 发行版之一，致力于创建一个的自由操作系统，Debian 的发展由开源社区维护

Debian 的下游发行版有：

  • Ubuntu

  • Deepin

    基于 Debian 的国产发行版，使用 Qt 技术开发了美观易用的交互界面，由开源社区维护

  • Kali

    由 Offensive Security 公司开发和维护，系统集成了大量的渗透和审计工具，囊括了大多数做安全测试的开源工具，是安全专家、黑客的首选发行版

  • Linux Mint

    Linux Mint 致力于创建一个开箱即用的操作系统，无需配置任何内容或安装额外的应用程序，默认配备了大多数人需要的应用程序

Arch

Arch 是一款基于 x86-64 架构的 Linux 发行版，采用滚动发行模式来更新系统，由开源社区维护

Arch 的下游发行版有：

  • Manjaro

NixOS

NixOS 采用独特方法来进行包管理和系统配置的工具，采用声明式的配置来构建系统

包管理器

Linux 有那么多发行版，那肯定有小伙伴会产生疑问：不同发行版之间的区别是什么？

其实所有的 Linux 发行版用的是同一个内核，它们的不同点之一是包管理，为了更好的理解包管理器，还得从早期的软件开发工作讲起...

编程与编译

计算机的 CPU 工作原理实际是对高低电平信号进行运算，映射成数字信号就是 0 和 1，也就是我们常说的二进制，所以 CPU 只能理解二进制数据指令

但是二进制数据指令对于人们来说是晦涩难懂的，为了更好地操作计算机，编程语言便诞生了，编程语言更加贴近于人类的语言系统

但是这也出现了一个问题，CPU 是理解不了人类语言的，它只认二进制数据指令，这时候就需要一个翻译官，将编程语言翻译成 CPU 能够识别的二进制数据指令，这便是编译器的工作，这个翻译过程被称为「编译」

开发人员用编程语言开发完一个程序后，就需要用编译器对源代码进行编译，编译的产物是一个存储二进制数据指令的压缩包（也被称为可执行文件），可执行文件能够被 CPU 理解并在操作系统中运行

这便是我们认知里的软件，比如 Windows 系统中的 .exe 文件就是可执行文件的一种

令人头疼的依赖

开发人员用编程语言开发一个程序时，并不是所有功能都从头自己实现，有时候某些功能需要借助其他开发者的程序实现，这就产生了依赖关系

比如程序 A 在开发过程中，需要借用程序 B 和程序 C 实现某些功能，就需要先下载好程序 B 和程序 C 的源代码进行编译，然后在程序 A 的开发中调用程序 B 和程序 C，我们就可以说程序 A 有 B、C 两个依赖

但是有个比较糟糕的情况是，这时候可能存在多级依赖关系，比如程序 B 有 D、E 两个依赖，而程序 D、程序 E 又又又依赖于其他程序...

为了能够开发程序 A，开发人员只能手动去处理这些依赖，把相关依赖的源代码都下载下来并一个个进行编译

包管理器的诞生

在早期时候，软件安装并不像现在那么方便，一个用户想要安装一个软件，则需要获得它的源代码后自行编译，这也意味着所有的依赖关系需要自行处理

于是人们思考着，有没有办法简化这些繁琐的过程，包管理器便诞生了

通俗易懂的来讲，包管理器便是现在各种软件商店的底层框架

包管理器本身也是一个程序，包管理器与软件仓库配合使用，人们先组织建立起一个远程软件仓库，开发者将程序开发完成后，按照指定格式打包上传到远程软件仓库中，这便有了一个软件集合

  • 用户想要安装某一个软件时，便可以与包管理器进行交互，告诉包管理器需要安装哪个软件，包管理器会从远程软件仓库拉取软件包到计算机中自动进行编译安装，并且它会检查这个软件所需要的依赖并自动处理，用户不再需要自己手动编译

  • 开发人员想要开发程序 A，需要 B、C 两个依赖，程序 B 有 D、E 两个依赖，而程序 E 又依赖于 F、G，那开发人员只需要用包管理器安装程序 B、程序 C 两个软件就行了，包管理器会自己处理所有依赖，这给开发者节省了大量的时间

包管理器还有其他功能，比如软件卸载、软件升级、安装指定版本的软件等等，给开发者和用户都带来了极大的便利

不同的包管理器

主流的 Linux 发行版为了软件生态，基本都创建了自己的远程软件仓库和包管理器

前面提到过，开发者开发完程序后，需要按照指定格式打包上传到远程软件仓库中，不同的发行版的打包要求不同，便出现了不同格式的软件包
